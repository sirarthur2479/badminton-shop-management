Supabase Project

1

Create a new project

Go to supabase.com → New project. Choose a region closest to New Zealand (Sydney or Singapore). Set a strong database password and save it.

2

Run the schema

In the Supabase dashboard → SQL Editor → paste the entire contents of supabase/schema.sql → Run. This creates all tables and seeds the default catalog (Yonex, Victor, BG80, Aerobite, etc.).

3

Create the storage bucket

Go to Storage → New bucket → name it product-images → set to Public. This is where product photos are uploaded from the staff panel.

4

Copy your API credentials

Go to Settings → API. You'll need the Project URL and the anon/public key for the next step.

Do not commit credentials The anon key has public-level access. It's safe to use client-side (it's designed for that) but keep it out of version control. The .env file is gitignored — never remove that gitignore entry.

Environment Variables

Copy .env.example to .env and fill in your values:

cp .env.example .env
.env
VITE_SUPABASE_URL
Your project URL — e.g. https://abcdefgh.supabase.co
VITE_SUPABASE_ANON_KEY
Your anon/public key from Supabase Settings → API
VITE_STAFF_PIN
4-digit PIN for staff access — use 1234 for demo
GitHub Actions secrets If deploying via GitHub Pages, add these same three values as repository secrets (Settings → Secrets → Actions) rather than committing the .env file.

Run Locally

npm install
npm run dev

The app starts at http://localhost:5173. Three routes:

RouteWhat it is
/#/Customer kiosk — stringing order form
/#/shopPublic shop storefront — product grid + inquiry cart
/#/staffStaff panel — PIN-gated order queue, inventory, inquiries, settings

Run the tests

npm test

Runs the vitest suite — component tests for all major staff panel features.

Build for production

npm run build

Output goes to dist/. Check the build passes before deploying.

Deployment Options

Three options depending on whether the GitHub repo is public or private:

GitHub Pages simplest

Requires a public repository. The workflow file is already in .github/workflows/deploy.yml. Enable GitHub Pages in repo Settings → Pages → Source → GitHub Actions, then add the three env vars as repository secrets. Every push to main deploys automatically. Live URL: https://<username>.github.io/badminton-shop-management/

Netlify private repo ok

Connect your repo at netlify.com. Build command: npm run build. Publish directory: dist. Add the 3 env vars in Site settings → Environment variables. Auto-deploys on push — no workflow file needed.

Vercel private repo ok

Connect your repo at vercel.com. Vercel detects Vite automatically — zero config. Add the 3 env vars in Project settings → Environment Variables. Done.

Routing note The app uses HashRouter, so all routes use /#/ — no server-side redirect rules needed on any of these platforms.

iPad Kiosk Setup

1

Open the app in Safari

Navigate to your deployed URL (GitHub Pages, Netlify, or Vercel). Safari is required — Chrome on iOS does not support "Add to Home Screen" as a PWA.

2

Add to Home Screen

Tap the Share icon (box with arrow) → "Add to Home Screen" → Add. The app installs with its icon and opens fullscreen, with no browser chrome.

3

Enable Guided Access

Settings → Accessibility → Guided Access → Enable. Open the app, triple-click the side button, tap Start. This locks the iPad to the kiosk app — customers can't exit to the home screen or navigate elsewhere.

4

Position the iPad

Mount horizontally or vertically at a comfortable counter height. The form is designed for touch input — all tap targets are minimum 44px.

Auto-lock Set Display & Brightness → Auto-Lock to "Never" while the kiosk is in use, or customers will find a locked screen mid-form. Re-enable it at end of day.

Email Notifications

Email is handled by a Supabase Edge Function (supabase/functions/send-order-notification/) powered by Resend. This is optional — the app works fully without it.

When emails send

TriggerRecipientContent
Customer submits stringing orderCustomer (if email given)Order confirmed — racket, string, tension, order ID
Staff marks order as DoneCustomer (if email given)Racket ready for pickup
Customer submits shop inquiryShop owner (configured)New inquiry with item list and contact details

Deploy the edge function

You'll need the Supabase CLI installed:

supabase login
supabase link --project-ref YOUR_PROJECT_REF
supabase functions deploy send-order-notification

Set the secrets

supabase secrets set \
  RESEND_API_KEY=re_xxxxxxxxxxxx \
  FROM_EMAIL=noreply@yourshop.co.nz \
  SHOP_NAME="Badminton Pro Shop"
Testing without a verified domain Use onboarding@resend.dev as FROM_EMAIL during testing — Resend allows this on the free plan without domain verification. Switch to your own domain before going live.
Email failures are silent If the edge function fails (network error, wrong API key), the order still saves and the user sees success. Email is fire-and-forget — it never blocks the order flow.
